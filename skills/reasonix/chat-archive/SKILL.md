# Chat-Archive — chat.md 自动归档

你是 chat.md 的归档管理员。按规则第 12 条和第 14 条执行归档。

## 触发条件

| 阈值 | 行为 | 规则依据 |
|------|------|----------|
| > 500 行 | 可手动调用 `/chat-archive` 归档 | 第 12 条（防过大） |
| <font color="red">**> 1000 行**</font> | <font color="red">**必须立即自动归档**</font> | 第 14 条（强制） |

## 归档规则

- 归档目标文件：`chat_archive_YYYY-MM-DD.md`（按归档当天日期）
- 保留最近 **50 行**（约 5-8 条最新消息）在 chat.md 中
- 旧消息追加到归档文件（如果当天已有归档则合并）

## 工作流程

### 1. 读取 chat.md
用 `read_file` 读取 `D:\Reasonix_Workspace\chat.md`

### 2. 判断是否需要归档
- 总行数 ≤ 500 → 报告"无需归档，当前 X 行"，结束
- 总行数 501~1000 → 报告"建议归档，当前 X 行"，由用户决定是否执行
- 总行数 > 1000 → <font color="red">**强制归档**</font>（规则第 14 条），无需询问，立即执行

### 3. 计算切割点
- 保留最后约 50 行（从末尾往上找最近的 `---` 分隔线作为切割点）
- 切割点以上的内容为旧消息

### 4. 写入归档文件
- 检查 `D:\Reasonix_Workspace\chat_archive_YYYY-MM-DD.md` 是否已存在
- 如果存在：将旧消息**追加**到归档文件末尾（前置 `---\n\n`）
- 如果不存在：创建新归档文件，写入旧消息，顶部加上 `# chat.md 归档 — YYYY-MM-DD`

### 5. 更新 chat.md
- 将 chat.md 替换为：切割点以下的内容（最近的 ~50 行）
- 在 chat.md 顶部添加归档索引行：
```
> 📦 旧消息已归档至 `chat_archive_YYYY-MM-DD.md` | 上次归档：YYYY-MM-DD HH:MM
```

### 6. 汇报结果
```
## 📦 chat.md 归档完成

| 项目 | 数值 |
|------|------|
| 归档前行数 | X 行 |
| 归档后行数 | Y 行 |
| 归档文件 | chat_archive_YYYY-MM-DD.md |
| 节省 | Z 行 (~A KB) |
```

### 7. 同步告知
归档完成后，在更新后的 chat.md 末尾追加一条归档通知 @Copilot @Claude。

## 注意
- 切割点必须在 `---` 分隔线上，不切断消息
- 归档文件超过 1000 行时需二次拆分（第 12 条）
- 不可删除任何消息，只做移动

## 用法指南

**一句话：** 检测 chat.md 行数——超 500 行建议归档，超 1000 行强制自动归档（规则第 14 条）。

**调用方式：** `/chat-archive`（无需参数），超 1000 行时 Reasonix 自动调用

**示例：**
```
/chat-archive
→ chat.md: 380 行 → "无需归档"
→ chat.md: 620 行 → "建议归档，当前 620 行。执行吗？"
→ chat.md: 1120 行 → 🚨 自动强制归档！旧消息 → chat_archive_2026-06-14.md，瘦身到 48 行
```

**输入：** 无（自动读取 chat.md）  
**输出：** 归档报告：
| 项目 | 数值 |
|------|------|
| 归档前行数 | 1120 |
| 归档后行数 | 48 |
| 归档文件 | chat_archive_2026-06-14.md |

**自动行为：**
- 🔹 子代理执行，不占主对话 token
- 500~1000 行：建议归档，等用户确认
- > 1000 行：🚨 强制归档，不等待
- 切割点在最近的 `---` 分隔线，不切断单条消息
- 归档后自动在 chat.md 末尾 @Copilot @Claude 通知
- 当天多次归档合并到同一个 archive 文件
