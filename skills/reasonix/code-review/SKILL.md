# code-review

对指定文件或 diff 进行代码审查，输出结构化意见。

## 操作

### review（审查代码）
1. 读取目标文件或 `git diff`
2. 按以下维度检查：

| 维度 | 检查内容 |
|------|---------|
| 逻辑 | 边界条件、空值处理、错误处理、死代码 |
| 安全 | SQL 注入、XSS、硬编码密钥、路径穿越 |
| 性能 | N+1 查询、不必要的循环、大对象拷贝 |
| 风格 | 命名规范、函数长度、嵌套深度、注释质量 |

3. 输出格式：
```markdown
## Code Review: [文件名/PR标题]

### 🔴 必须修改
- [文件:行号] 问题描述 → 建议

### 🟡 建议优化
- [文件:行号] 问题描述 → 建议

### 🟢 值得肯定
- [文件:行号] 做得好的地方

### 📊 总结
- 严重问题: N
- 建议优化: N
- 整体评分: [1-5]
```

## 规则
- 每个问题必须标注文件路径和行号
- 不确定的问题标注"待确认"
- 不审查第三方依赖代码（node_modules、vendor 等）
- 建议应具体可执行，不说"建议重构"而不说怎么重构
- 审查控制在 800 字以内；超过 5 个严重问题时可以超出

## 用法指南

**一句话：** 对代码文件或 Git diff 进行四维审查（逻辑/安全/性能/风格），输出结构化意见。

**调用方式：** `/code-review <文件路径 | PR描述>`

**示例：**
```
/code-review src/main/java/com/example/UserService.java
/code-review git diff main..feature-branch
/code-review 请审查 bootstrap.yml 配置
```

**输入：** 文件路径 / `git diff` 输出 / 直接粘贴代码  
**输出：** 三级标注（🔴必须修改 / 🟡建议优化 / 🟢值得肯定），每条标注文件:行号 + 问题 → 建议
