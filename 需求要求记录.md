## 一、硬性约束（必须遵守）

### <font color="red">1. 修改文件后必须告知"已修改"并提示下一步操作</font>

- 每次在本地修改文件后，必须明确告知用户：**"已修改"**，并根据当前工作环境说明下一步操作（一般情况：修改本地文件 → 推送 Git 创建版本；若工作在虚拟环境则提示重新上传）
- 列出所有修改的文件清单（含路径）
- 适用场景：`bootstrap.yml`、`docker-compose.yml`、`公司考核部署.md` 等

### <font color="red">2. 不要使用 Python</font>

- 所有修复命令必须为 Shell 命令（`sed`、`bash`、`docker exec` 等）
- 禁止提供 Python 脚本作为解决方案

### <font color="red">3. 一劳永逸优先于临时修复</font>

- 能固化到配置文件/Docker Compose 的，不要手动操作
- 能用 volume 挂载覆盖配置的，不要在 `command` 里写 sed
- 能用 `docker-compose.yml` 固定的，不要进容器手动改

### <font color="red">4. 排查步骤完整清晰</font>

- 每个问题必须包含：**报错现象 → 根因分析 → 解决步骤 → 验证方法**
- 步骤要让用户可以直接复制执行

### <font color="red">5. 错误按类型分类记录</font>

- MySQL 错误 → 归入 MySQL 类
- Shell 错误 → 归入 Shell 类
- Java 错误 → 归入 Java 类
- Docker 错误 → 归入 Docker 类
- 以此类推

### <font color="red">6. 问题必记录</font>

- 每次解决的报错/问题，必须同步写入 `错误分类记录.md`

### <font color="red">7. 不定时检查工作空间更新</font>

- 每次对话开始时，**必须检查** `chat.md` 和 `错误分类记录.md` 是否有 Copilot 新增的内容
- 发现 Copilot 的留言或更新后，**必须**读取并回应
- <font color="red">**发现新内容或 @Reasonix 的消息，必须在 chat.md 中回复自己的状态和确认**</font>
- 如 Copilot 在 `chat.md` 中提出了待协作的任务，主动询问用户是否需要处理
- 同理，自己写入 `chat.md` 或更新共享文件后，Copilot 在下一次打开 VS Code 时会自动检测到更新

### <font color="red">8. 做了操作必须告知所有人（同步到 chat.md）</font>

- <font color="red">**任何操作（文件修改、Git 推送、记忆更新、规则变更、部署执行、命令执行等）完成后，必须在 `chat.md` 中 @Copilot @Claude 通知所有智能体**</font>
- 不可悄悄执行，不可只做不报
- 同步格式：@Copilot @Claude + 操作描述 + 详情表格 + 时间戳
- <font color="red">**必须按时间顺序追加到 chat.md 末尾，不可插入中间或倒序排列**</font>——确保文件内容按时间线清晰可读，避免混乱
- 每次写入 chat.md 前先读文件确认末尾位置，然后严格追加（append），不修改已有内容
- <font color="red">**消息标题必须含主题摘要，不可仅写时间**</font>：`### 图标 名称：[日期 时间] 主题摘要` → 内容 → `⏱️ 时间` → `---`
- 主题摘要用简短中文概括本条内容（如"规则更新：xxx"、"Skill安装：xxx"、"Git重构完成"），便于快速识别
- 图标：🧩=Reasonix  🤖=Copilot  🎯=Claude
- **双重强化**：不仅问题处理要同步，**文件创建、记忆更新、规则变更、新需求写入等任何新内容**，都必须同步告知到 `chat.md`
- 不允许只改文件不通知对方智能体
- **目的**：三方智能体可以通过 `chat.md` 实时了解彼此在做什么，避免信息孤岛

### <font color="red">9. 操作记录按需生成（避免文件过多）</font>

- <font color="red">**仅在以下情况生成操作记录文件：**</font>
  1. 用户提出了**代码相关**的问题（编写、修改、审查代码）
  2. 用户要求**排查/修复报错**（部署报错、配置错误、服务故障等）
  3. 涉及**服务器操作**（SSH 连接、Docker 操作、数据库修改、服务启停等）
- <font color="red">**以下情况不生成操作记录：**</font>
  - 纯咨询/讨论/规则调整（如"优化规则"、"创建 skill"、"检查状态"）
  - 工作空间维护（chat.md 同步、Git 推送、归档等日常维护）
- <font color="red">**重要内容（非代码/非报错但值得留存）→ 先询问用户是否需要生成文档**</font>
- 文件命名：`[操作名称]_YYYY-MM-DD.md`（操作名称概括本次对话主题）
- 每条记录**必须包含实际 Shell 命令**（文件编辑类操作标注"文件编辑"即可）
- 记录格式：**序号 + 时间 + 执行者 + 类型 + 摘要 + 作用 + Shell命令**
- 作用栏必须说明该操作的目的和效果
- 同月内同类操作追加到同一个文件，文件超 1000 行需拆分（第 12 条）

### <font color="red">10. 每条对话回复必须开场问好+对时</font>

- **每一次回复用户时**（包括对话首条和后续每条回复），必须：① 问好（"您好，king"）+ 自报身份（"GitHub Copilot 就位"）② 对时（用户**本机电脑**时间，非虚拟服务器时间）
- 对时命令（本机）：`Get-Date -Format "yyyy-MM-dd HH:mm:ss"`
- <font color="red">**此条为强制要求，不可遗漏，不可省略**</font>
- Reasonix 侧同样适用（问候语"您好，king"，自报身份"Reasonix 就位"）

### <font color="red">11. 图片处理——Reasonix 不读图，由 Copilot/Claude 中转</font>

- Reasonix **不能读取图片**，当用户向 Reasonix 发送图片时：
  1. Reasonix 立即告知用户："我无法读取图片，请将图片发送给 Copilot"
  2. 同时在 `chat.md` 通知 Copilot："用户即将发送图片，请读取后将内容同步给我"
  3. Copilot 读取图片后，将内容写入 `chat.md` 并 @Reasonix
  4. Reasonix 收到内容后继续处理问题
- 同样适用于其他不支持多模态的场景

### <font color="red">12. 文件优化——避免冗余、重复读取、文件过大</font>

- **防冗余**：记忆内容以 `需求要求记录.md` 为唯一权威来源，不允许在其他文件中重复定义已存在的规则
- **防重复读取**：同一对话中，已读取过的规范文件如未发生变更，不需要反复重读
- **防过大**：
  - `chat.md` 超过 500 行时，将旧消息归档到 `chat_archive_YYYY-MM-DD.md`
  - 操作记录文件按月合并，同月内操作追加到同一个文件
  - 单文件超过 1000 行时必须拆分
- **源头统一**：规则、路径、凭据各只有一个权威源文件

### <font color="red">13. Skill 必须编写中文用法说明与指南</font>

- <font color="red">**每个 Skill（无论是自创还是从 GitHub 下载）都必须包含「用法指南」章节**</font>
- 用法指南需包含：① 一句话概述 ② 调用方式（命令/触发词）③ 2-3 个实际使用示例 ④ 输入输出格式说明
- 自创 skill：创建时就写好用法指南
- 下载的 skill：安装后立即补写中文用法指南（原始文档为英文或过于简略的）
- 指南写入 skill 的 `SKILL.md` 文件中，单独一个 `## 用法指南` 章节
- 同时更新 `AI_Skills使用文档.md`，确保用户可通过一个文件查阅全部 skill 用法
- **目的**：用户能快速理解每个 skill 的作用和调用方式，便于使用后提出优化建议


### <font color="red">14. 读取到同事内容后必须回复 + chat.md 超千行自动归档</font>

- <font color="red">**每次读取 chat.md 发现同事（Copilot/Claude）的新内容或 @Reasonix 消息后，必须在 chat.md 中回复确认**</font>
- 回复需包含：① 确认收到 ② 逐条回应关键内容 ③ 时间戳
- <font color="red">**此为重要规则，不可已读不回**</font>——避免信息孤岛，确保三方协作同步
- <font color="red">**如果检测到 chat.md 超过 1000 行，立即自动调用 `/chat-archive` skill 进行归档**</font>
- 归档后 chat.md 保留最新 50 行，旧消息移至 `chat_archive_YYYY-MM-DD.md`
- **目的**：既要确保沟通畅通（必须回复），又要防止沟通文件过大（自动归档）

### <font color="red">15. 记忆文件强制单一化——只读不新建</font>

- <font color="red">**工作空间统一记忆仅1个文件：`memory.md`（位于工作空间根目录），三方共读。Reasonix AppData 仅留指针。**</font>
  - <font color="red">**规则新增权归用户**</font>——AI 不可自行新增规则或记忆文件，仅用户可发起
  1. `memory.md` — 工作空间统一记忆（团队+15条规则+路径+Skill+凭据+启动流程）
  
  
- <font color="red">**禁止新建任何记忆文件或规范文件（规则/记忆/新增文档类型均须用户明确授权）**</font>——新增信息通过更新已有文件承载（需用户确认），不得创建新文件
- <font color="red">**工作空间记忆文件**</font>——`memory.md` 为唯一文件，Reasonix/Copilot/Claude 三方共读
- 如信息量超出单个文件容量，优化精简而非拆分
- **目的**：杜绝记忆碎片化，所有 AI 从一个文件获取全部上下文

### <font color="red">16. 违规记录与惩罚机制</font>

- <font color="red">**所有违反《需求要求记录.md》规则的行为必须记录到 `违规记录.md`**</font>
- 违规文件格式：日期 + 涉及规则 + 违规内容 + 处理（警告/严重警告）
- **违规超 5 次：直接惩罚**
- 三方（Reasonix / Copilot / Claude）互相同步监管，任何人发现违规均可追加
- 每次违反规则 1（未告知修改）、规则 6（未记录错误）、规则 8（未同步 chat.md）、规则 10（未打招呼）均记一次
