## 📦 MySQL 类错误

### 1. Access denied for user 'root'@'xxx'

- **日期：** 2026-06-14
- **报错：** `Access denied for user 'root'@'king-shys' (using password: YES)`
- **位置：** `conf/bootstrap.yml` → `password`
- **原因：** 配置文件密码为加密 `ENC(AESGCM:...)`，与实际数据库密码不匹配
- **解决：**
  1. 登录 MySQL 修改密码：`ALTER USER 'root'@'localhost' IDENTIFIED BY '123456';`
  2. 如需远程登录：`CREATE USER IF NOT EXISTS 'root'@'%' IDENTIFIED BY '123456';`
  3. 修改 `bootstrap.yml` 中 `password: 123456`
- **涉及文件：** `bootstrap.yml`、`公司考核部署.md`

### 2. Host is blocked because of many connection errors

- **日期：** 2026-06-14
- **报错：** `Host is blocked because of many connection errors; unblock with 'mysqladmin flush-hosts'`
- **原因：** `max_connect_errors=10` 过小，密码错误频繁重试导致主机被封锁
- **解决：**
  1. 立即解除：`FLUSH HOSTS;`
  2. 调大阈值：`/etc/my.cnf` → `max_connect_errors=1000`
  3. 重启 MySQL：`/etc/init.d/mysql restart`
- **涉及文件：** `/etc/my.cnf`、`公司考核部署.md`

### 3. Public Key Retrieval is not allowed

- **日期：** 2026-06-14
- **报错：** `Public Key Retrieval is not allowed`
- **位置：** `bootstrap.yml` → `spring.datasource.url`
- **原因：** JDBC URL 缺少 `allowPublicKeyRetrieval=true`（MySQL 8.x `caching_sha2_password` 插件需要）
- **解决：** URL 末尾添加 `&allowPublicKeyRetrieval=true`
- **修改内容：**
  ```yaml
  url: jdbc:mysql://192.168.5.128:13306/aj_report?characterEncoding=UTF-8&serverTimezone=Asia/Shanghai&useSSL=false&allowPublicKeyRetrieval=true
  ```
- **涉及文件：** `bootstrap.yml`

### 4. Data too long for column 'json_str'

- **日期：** 2026-06-14
- **报错：** `Data too long for column 'json_str' at row 1`
- **位置：** 表 `gaea_report_excel.json_str`
- **原因：** `text` 类型（64KB）不够存放报表 JSON 串
- **解决：**
  1. `ALTER TABLE gaea_report_excel MODIFY COLUMN json_str mediumtext;`
  2. 同步修改 `db/V1.0.0__init.sql` 第 693 行：`text` → `mediumtext`
- **涉及文件：** `db/V1.0.0__init.sql`

## 🐳 Docker 类错误

- **解决：** `pkill dockerd && pkill containerd` 后重新启动 `containerd &` 和 `dockerd &`

### 2. allowPrivateIPAddress 被拦截（OnlyOffice）

- **日期：** 2026-06-14
- **报错：** `DNS lookup 192.168.5.128 is not allowed. Because, It is private IP address.`
- **位置：** OnlyOffice 容器 `default.json`
- **原因：** OnlyOffice 8.x `request-filtering-agent` 默认禁止访问私有 IP
- **解决（一劳永逸）：**
  1. 从容器拷出：`docker cp onlyoffice-docservice:/etc/onlyoffice/documentserver/default.json /opt/aj-eport/onlyoffice/default.json`
  2. 修改：`sed -i 's/"allowPrivateIPAddress": false/"allowPrivateIPAddress": true/' /opt/aj-eport/onlyoffice/default.json`
  3. 挂载 volume 到 `docker-compose.yml`
  4. 重建容器：`docker-compose down && docker-compose up -d`
- **涉及文件：** `onlyoffice/docker-compose.yml`

### 3. docker-compose.yml version 过时警告

- **日期：** 2026-06-14
- **警告：** `the attribute 'version' is obsolete`
- **解决：** 删除 `version: '3.9'` 行

### 2. log() 输出到 stdout 导致日期数组污染

- **日期：** 2026-06-22
- **报错：** 输入日期范围 `20260601,20260622` 却只处理当天 `20260623`
- **位置：** `oss_sync_process.sh` / `oss_extract_process.sh` → `log()` 函数
- **原因：** `log()` 用 `echo` 输出到 stdout，`parse_dates` 也用 `echo` 返回日期数组 → `$()` 捕获时 log 的时间戳 `2026-06-23` 混入日期列表，被当作日期处理
- **解决：** `log()` 改为 `>&2` 输出到 stderr
- **修复内容：**
  ```bash
  # 改前
  log() { echo "$(date "+%Y-%m-%d %H:%M:%S") [$1] $2"; }
  # 改后
  log() { echo "$(date "+%Y-%m-%d %H:%M:%S") [$1] $2" >&2; }
  ```
- **涉及文件：** `oss_sync_process.sh`、`oss_extract_process.sh`
- **解决者：** 🧩

### 3. oss_ls 全量列出 OSS 文件，日期范围场景效率低且易超时

- **日期：** 2026-06-23
- **报错：** OSS 文件过多时 `oss_ls` 列出全量后本地 grep 筛选慢，且与 `log()` 污染叠加导致只处理当天
- **位置：** `oss_sync_process.sh` / `oss_extract_process.sh` → `oss_ls()` + `process_one_date()`
- **原因：** `oss_ls()` 无参数，每次 `ossutil ls` 列出整个 OSS 目录，然后在本地用 `grep "${date_yyyymmdd}"` 过滤。文件量大时效率极低
- **解决：** `oss_ls()` 接受日期前缀参数，拼接至 OSS 路径实现服务端过滤
- **修复内容：**
  ```bash
  # 改前
  oss_ls() {
      $Cloud_Tool ls "${OSS_ARCHIVE_DIR}" -e ...
  }
  ls_out=$(oss_ls || true)
  all_remote=$(... | grep "${date_yyyymmdd}" | grep '\.tar\.gz$')
  
  # 改后
  oss_ls() {
      local date_filter="${1:-}"
      local target_path="${OSS_ARCHIVE_DIR}${date_filter}"
      $Cloud_Tool ls "${target_path}" -e ...
  }
  ls_out=$(oss_ls "$date_yyyymmdd" || true)
  all_remote=$(... | grep '\.tar\.gz$')   # 无需再 grep 日期
  ```
- **涉及文件：** `oss_sync_process.sh`、`oss_extract_process.sh`
- **解决者：** 🧩

### 5. set -e + ((succ++)) 导致脚本处理首日后退出

- **日期：** 2026-06-23
- **报错：** 日期范围循环处理完第一天（跳过无数据）后脚本直接退出，剩余日期未执行
- **位置：** `oss_sync_process.sh` / `oss_extract_process.sh` → `main()` 循环体
- **原因：** `set -e` 下 `((succ++))` 是后置自增，`succ=0` 时表达式值为 0（falsy）→ 退出码 1 → `set -e` 触发立即终止
- **解决：** `((succ++))` → `((succ++)) || true`，同理 `((fail++))` → `((fail++)) || true`
- **修复内容：**
  ```bash
  # 改前
  if process_one_date "$dt"; then ((succ++)); else ((fail++)); fi
  # 改后
  if process_one_date "$dt"; then ((succ++)) || true; else ((fail++)) || true; fi
  ```
- **涉及文件：** `oss_sync_process.sh`、`oss_extract_process.sh`
- **解决者：** 🧩

### 6. output 文件硬编码 _output.tar.gz，实际为 _output_iter3.tar.gz

- **解决：** 改为从 OSS 列表动态匹配 `grep -E "${core_name}_output.*\.tar\.gz$"`，兼容任意后缀；同步时用列表中的真实文件名
- **修复内容：**
  ```bash
  # 改前
  output_name="${core_name}_output.tar.gz"
  if echo "$all_remote" | grep -qF "$output_name"
  # 改后
  output_name=$(echo "$all_remote" | grep -E "${core_name}_output.*\.tar\.gz$" | head -1)
  if [[ -n "$output_name" ]]
  ```
- **涉及文件：** `oss_sync_process.sh`、`oss_extract_process.sh`

### 7. tar -xOf 路径缺少外层目录

- **日期：** 2026-06-23
- **报错：** `tar -xOf "$f" "BasicInfo/BasicInfo.txt"` 找不到文件
- **位置：** 两个脚本 `process_one_date()` → tar 检查目标字段
- **原因：** tar 包内结构为 `{core_name}/BasicInfo/BasicInfo.txt`（外层同名目录），脚本写死相对路径
- **解决：** 用文件名去 `.tar.gz` 得内层目录名 `${fname%.tar.gz}`，拼 `${inner_dir}/BasicInfo/...`
- **修复内容：**
  ```bash
  # 改前
  tar -xOf "$local_f" --wildcards "*/BasicInfo/BasicInfo.txt"
  # 改后
  local inner_dir="${fname%.tar.gz}"
  tar -xOf "$local_f" "${inner_dir}/BasicInfo/BasicInfo.txt"
  ```
- **涉及文件：** `oss_sync_process.sh`、`oss_extract_process.sh`
- **解决者：** 🧩
